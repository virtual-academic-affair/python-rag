1. TONG QUAN & MUC DICH (MISSION STATEMENT)
================================================================================

AI Service là một FastAPI microservice hiệu năng cao, đóng vai trò là \"Hệ điều hành AI\" trung tâm cho các nghiệp vụ giáo vụ. Dự án được thiết kế để xử lý các tài liệu học vụ phức tạp và tự động hóa quy trình phản hồi email từ sinh viên/giảng viên.

HỆ THỐNG GIẢI QUYẾT 2 BÀI TOÁN LỚN:
1.  EMAIL TRIAGE & AUTOMATION: Tự động phân loại hàng nghìn email đổ về từ NestJS, trích xuất dữ liệu đăng ký môn học hoặc tự động soạn thảo bản thảo phản hồi (Email Draft) dựa trên kho tài liệu thực tế của nhà trường.
2.  MODULAR RAG DOCUMENT BRAIN: Xây dựng một \"bộ não\" kiến thức từ hàng nghìn văn bản quy định (.pdf, .md, .docx). Hệ thống không chỉ tìm kiếm từ khóa mà hiểu được \"Mục lục\" và \"Cấu trúc\" tài liệu để trả lời chính xác từng trang, từng dòng.

2. KIEN TRUC HE THONG MODULAR RAG
================================================================================

Dự án sử dụng kiến trúc RAG Modular (phi tập trung), kiểm soát hoàn toàn dữ liệu với luồng pipeline mới (State-free):

SƠ ĐỒ LUỒNG PHỐI HỢP COMPONENT (Upload -> R2 -> LlamaParse -> PageIndex (TOC) -> Corpus Tree):
1.  UPLOAD LAYER: Nhận file -> Validate Metadata (Required fields) -> R2 (Lưu file gốc) -> MongoDB (Status: uploading).
    - [NEW]: Hỗ trợ clientId để đẩy thông báo tiến trình real-time qua WebSocket.
    - [NEW]: Mọi API Multipart/Form-data đều hỗ trợ song song `snake_case` và `camelCase` cho các trường dữ liệu đầu vào.
2.  INDEXING LAYER (ASYNC):
    - LlamaParse: Trình phân tích PDF tốt nhất hiện nay, xử lý được bảng biểu (tables) trong quy chế.
    - PageIndex: Đóng vai trò kiến trúc sư mục lục. Nó không lấy text thô mà xây dựng một cây Phân cấp (Building an Hierarchical Tree).
    - Corpus Indexer: Chịu trách nhiệm phân loại tài liệu vào Corpus Topic Tree và lưu trữ vào MongoDB.
3.  RAG LAYER:
    - Khi có Query: Hệ thống duyệt cây Corpus Topic Tree để lọc ra các tài liệu và FAQ liên quan nhất.
    - Corpus Traversal: Trả về danh sách File Candidates và Supporting FAQs.
    - MongoDB: Trả về "doc_description" (1 câu tóm tắt file) để AI định hướng context.
4.  AGENTIC CHAT LAYER:
    - [NEW] RAG Gate: Hệ thống sử dụng AI để đánh giá câu hỏi. Nếu là câu hỏi xã giao (Small Talk), AI sẽ trả lời trực tiếp mà không cần gọi RAG, giúp tiết kiệm tài nguyên.
    - AI Agent (Gemini) không nhận hàng tấn text. Nó nhận danh sách File IDs.
    - Agent sử dụng bộ công cụ (Tools) để gọi lại PageIndex: \"Hãy cho tôi xem mục lục File ID 1\" -> \"Tôi muốn đọc trang 5 của File ID 1\".

5. LUONG XU LY EMAIL (RABBITMQ INGEST ENGINE)
================================================================================

Đầu vào từ RabbitMQ (Payload IngestMessage):
  - field: data.messageId (int)
  - field: data.subject (str)
  - field: data.content (str)
  - field: data.senderEmail/senderName (str)

LUỒNG XỬ LÝ NỘI BỘ (EmailIngestConsumer.py):
1.  Nhận message từ queue -> Ack ngay lập tức nếu JSON lỗi (Anti-Poison).
2.  Validate mảng `data` qua Pydantic.
3.  Dispatcher: Gọi `EmailWorkflowOrchestrator.process_request`.
4.  Async Execution: Sử dụng `asyncio.run_coroutine_threadsafe` gửi coroutine vào loop chính của FastAPI.
5.  Timeout Control: Mỗi email có tối đa 120 giây xử lý trước khi bị ngắt.
6.  Final ACK/NACK:
    - OK -> Xóa khỏi queue.
    - Error (Database/LLM sập) -> Requeue=True để chờ hệ thống hồi phục.

6. LABELCLASSIFIERSERVICE (AI TRIAGE)
================================================================================

6.1. CHI TIẾT PROMPT PHÂN LOẠI (SYSTEM PROMPT):
AI nhận vai trò \"Senior Triage Assistant\" với các chỉ dẫn nghiêm ngặt:
- classRegistration: Yêu cầu hành động về học phần (đăng ký, hủy, xin mở lớp). Keyword: \"mã học phần\", \"học phần\", \"mở lớp\".
- task: Yêu cầu từ đơn vị nội bộ (hành chính, khoa) giao việc cho phòng giáo vụ.
- inquiry: SV hỏi đáp về quy định (QA). Không yêu cầu hành động đăng ký trực tiếp.
- other: Spam, lời chào, nội dung không liên quan.

RÀNG BUỘC CỦA PROMPT:
- Trả về DUY NHẤT một token nhãn chuẩn.
- Không giải thích, không markdown.
- Ưu tiên: Registration > Task > Inquiry > Other.

6.2. THUẬT TOÁN NORMALIZE & OVERRIDES:
- AI thường trả về \"classRegistration.\" (có chấm) hoặc \"Label: task\".
- Logic Normalize (Python regex) sẽ bóc tách chỉ giữ lại Alphabet, chuyển về lowercase và map vào bộ alias chuẩn.
- Override Heuristic: Nếu AI trả về 'other' nhưng trong email chứa từ khóa \"đăng ký học phần\", hệ thống sẽ ép kiểu về `classRegistration` để đảm bảo không bỏ sót yêu cầu của SV.

7. LLM GATEWAY (LITELLM)
================================================================================

Dự án dùng `LLMGateway` in-process trên LiteLLM Python SDK cho toàn bộ LLM call application-owned:
- Chat/email query analysis và direct reply.
- Corpus topic assignment và Corpus traversal.
- FAQ answering.
- PageIndex answering agent (non-stream và stream).
- Email classification, mixed-intent split và payload extraction.
- FAQ candidate synthesis.

Contract nội bộ dùng message/tool-call format trung lập provider; LiteLLM model phải có prefix,
ví dụ `gemini/gemini-2.5-flash`. Credential của provider active được cấu hình một lần qua `LLM_API_KEY`.
`app/integrations/pageindex/**` là integration bên ngoài và không đi qua gateway này, nhưng dùng chung
`LLM_MODEL` để tránh một alias model riêng cho PageIndex. Project chỉ dùng gateway chung cho
application-owned LLM calls, không còn prompt framework hoặc direct Gemini wrapper.

7.1. RETRY, TIMEOUT VÀ USAGE:
- Gateway là nơi duy nhất retry application LLM calls; LiteLLM internal retry bị tắt để tránh retry nhân đôi.
- Timeout dùng chung qua `LLM_TIMEOUT_SECONDS`.
- Usage được chuẩn hóa thành `prompt_tokens`, `completion_tokens`, `total_tokens`.
- Streaming ghép các tool-call argument delta hoàn chỉnh trước khi validate và chạy tool.

7.2. MODEL SETTINGS:
- `LLM_MODEL`: model dùng chung cho application LLM calls và các LLM call nội bộ của PageIndex.
- `LLM_DETERMINISTIC_TEMPERATURE=0.0`: structured output, classification/extraction, FAQ answer/synthesis và tool agents.
- `LLM_DIRECT_REPLY_TEMPERATURE=0.5`: direct conversational reply khi không cần RAG.
- System prompt, response format, tools, reasoning và max turns là behavior từng use case, không phải model config chung.
- Các ngưỡng xây cây như `PAGEINDEX_MAX_TOKEN_NUM_EACH_NODE` và giới hạn vòng lặp như
  `PAGEINDEX_AGENT_MAX_TURNS` vẫn là config riêng vì chúng điều khiển thuật toán, không phải sampling của model.

8. HE THONG WORKFLOWS & GRPC INTEGRATION
================================================================================

Dự án giao tiếp với Nest-api thông qua một `GrpcClient` singleton (app/integrations/grpc/client.py).

8.1. CLASSREGISTRATION SERVICE:
- Logic: AI nhận email -> Trích xuất mảng JSON items (action, subjectCode, className...).
- Mapping: Tự động chuẩn hóa `messageId` từ email ban đầu.
- gRPC Call: Gọi `rpc Create` của `ClassRegistrationServiceStub`.
- Payload: Chứa đầy đủ các trường Pydantic của NestJS để tạo Entry hỗ trợ SV.

8.2. INQUIRY SERVICE:
- B1: `EmailQueryAnalyzerService` trích xuất câu hỏi chuẩn, inquiry types và metadata filter từ subject/body email.
- B2: `InquiryService` gọi shared `RagQueryPipeline.answer_email(...)` với behavior email: không dùng chat analyzer và không direct reply.
- B3: Pipeline chạy Corpus Tree traversal, hydrate/rerank FAQ trước rồi cho LLM đọc FAQ. Nếu FAQ đủ bao phủ toàn bộ câu hỏi, trả lời source=`faq`; nếu không đủ mới hydrate/rerank file và chạy PageIndex agent.
- B4: Gemini/PageIndex soạn câu trả lời email Markdown. Trích dẫn `(^...)` luôn được resolve sang document viewer bằng `DOCUMENT_VIEW_URL_PREFIX + fileId`. Nếu `to_rich_text=True`, email layer convert Markdown sang HTML.
- gRPC Call: Gọi `rpc Create` của `InquiryServiceStub` để ghi nhận thắc mắc và lưu bản thảo email.

8.3. TASK SERVICE:
- Logic: AI trích xuất `name`, `description`, `due` (deadline) và `assignee_keywords` (ví dụ: "cô Lan").
- Resolver: Chạy song song (`asyncio.gather`) các cuộc gọi gRPC `FindOneByKeyword` tới `AuthServiceStub` để lấy User ID thực tế.
- gRPC Call: Gọi `rpc Create` của `TaskServiceStub` để tạo task và gán người thực hiện.

9. EMAIL DRAFT SERVICE (RAG-BASED REPLY)

CẤU TRÚC PROMPT SOẠN EMAIL (EMAIL_SYSTEM_PROMPT):

- Vai trò: Tư vấn viên chính thức của Phòng Giáo vụ trường đại học.
- Quy tắc bắt buộc:
  1. `BASE_PAGEINDEX_SYSTEM_PROMPT` giữ quy trình tool, nguyên tắc nguồn, định dạng answer và citation dùng chung; `CHAT_SYSTEM_PROMPT`/`EMAIL_SYSTEM_PROMPT` chỉ thêm persona, cách xưng hô và quy tắc trả lời.
  2. Mọi tool reasoning công khai là một câu tiếng Việt ngắn; không yêu cầu raw chain-of-thought.
  3. Luôn xem cấu trúc bằng `get_document_structure`, sau đó bắt buộc đọc nội dung thật bằng `get_page_content` trước khi kết luận.
  4. FAQ context chỉ hỗ trợ hiểu ý định khi FAQ chưa đủ trả lời; không dùng FAQ làm nguồn quy định hoặc citation PageIndex.
  5. Câu trả lời Markdown được bọc trong `<answer>...</answer>`; không chào hỏi, không lộ file ID/tool, và đặt citation một lần sau phần nội dung dùng xong một mục tài liệu.
  6. Answering model vẫn nhận subject/body gốc làm ngữ cảnh cùng câu hỏi đã chuẩn hóa, metadata, FAQ hỗ trợ và tài liệu; instruction chỉ yêu cầu trả lời trực tiếp, không yêu cầu soạn thảo email.
  7. Citation `(^Tên mục lục tương ứng)` được resolver chuyển thành link document viewer.

12. FAQ MODULE V2 (SEMANTIC FAQ & AUTO-SYNTHESIS)
================================================================================

Module FAQ là kho tri thức hỏi đáp có quản trị CRUD/import và được dùng trong shared RAG pipeline như một nguồn context ưu tiên. Pipeline không trả nguyên văn FAQ answer; LLM phải đọc một hoặc nhiều FAQ liên quan và tự tổng hợp câu trả lời Markdown.

11.1. MANUAL FAQ MANAGEMENT (Quản lý thủ công):
- Hỗ trợ thao tác CRUD (Tạo, Sửa, Xóa) cho FAQ.
- FAQ Answering: LLM đọc nội dung FAQ liên quan để trả lời trực tiếp khi FAQ đủ bao phủ toàn bộ câu hỏi.
- Danh sách FAQ giờ đây được sắp xếp mặc định theo `createdAt` (giảm dần).
- Metadata Filter: Hỗ trợ lọc theo `academicYear` và `cohort`. Nếu mảng rỗng (empty array), FAQ áp dụng cho toàn bộ sinh viên.
- Bulk Management: 
  - Hỗ trợ Import hàng loạt từ file Excel (.xlsx) với cơ chế map cột linh hoạt (theo tên hoặc ký tự).
  - [FORMATTING]: Hệ thống tự động bảo toàn định dạng Rich Text (In đậm, In nghiêng, Gạch chân) và các Hyperlink từ Excel, chuyển đổi chúng sang định dạng Markdown chuẩn.
  - Kỹ thuật: Sử dụng `openpyxl` với tham số `rich_text=True` để truy cập các fragments văn bản và thuộc tính font, kết hợp xử lý `cell.hyperlink`.
  - Tự động bóc tách Metadata từ các cột Excel chỉ định.
  - Tối ưu hiệu suất: Sử dụng xử lý hàng loạt và lưu trữ trực tiếp vào MongoDB.
  - Cơ chế Skip & Continue: Tự động bỏ qua câu hỏi trùng lặp và ghi nhận chi tiết các dòng lỗi mà không làm gián đoạn tiến trình import chung.
  - Hỗ trợ API Bulk Create qua JSON để tích hợp với các hệ thống bên ngoài.

11.2. INTERACTION LOGGING (Theo dõi tương tác):
- Hệ thống tự động ghi lại lịch sử Chat và Email Inquiry thành `InteractionLogs`.
- Deduplication: Tự động lọc trùng lặp nếu câu hỏi (unaccented) xuất hiện trong vòng 24 giờ.
- TTL Index: Lịch sử tự động xóa sau 90 ngày (cấu hình qua `FAQ_LOG_RETENTION_DAYS`) nhờ MongoDB TTL.

11.3. AUTO FAQ SYNTHESIS (Tự động tổng hợp FAQ):
- Model, repository và DTO cho FAQ Candidate vẫn còn trong codebase.
- Endpoint `POST /api/faqs/synthesis` hiện trả `501` với thông báo synthesis tạm disabled pending architecture migration.
- Admin review candidate vẫn được giữ cho dữ liệu/candidate hiện có, nhưng không nên coi synthesis background job là luồng production đang chạy.

11.4. FORMS MANAGEMENT MODULE (Quản lý biểu mẫu):
- Module này quản lý danh sách các liên kết biểu mẫu và quy trình học thuật, giúp sinh viên/giảng viên truy cập nhanh các mẫu đơn cần thiết.
- **Unified Schema**: Gộp tên hiển thị và đường dẫn thành một trường Rich Text (HTML), cho phép AI và người dùng linh hoạt trong việc trình bày.
- **Link Processing**: Mọi liên kết trong Rich Text được tự động xử lý bởi Frontend (hàm `fixRichTextLinks`) để mở trong tab mới, đảm bảo tính nhất quán với module FAQ.
- **Bulk Import**: Nạp hàng loạt biểu mẫu từ Excel/CSV với cơ chế chuẩn hóa số liệu (ví dụ: `2020.0` -> `2020`) và bảo toàn định dạng Rich Text.
- **Shared Excel Integration**: Sử dụng module `app/integrations/excel/utils.py` để dùng chung logic tìm kiếm cột cho cả Excel và CSV.

11.4. FAQ SEARCH & ANSWERING STRATEGY (Tìm kiếm và trả lời từ FAQ):
- Admin/list API vẫn hỗ trợ `search` text và metadata filter để quản lý FAQ.
- Trong RAG query, FAQ docs được lấy từ Corpus Tree retrieval theo chủ đề và metadata/user role, sau đó rerank riêng bằng Cohere.
- `FaqAnswerService` gọi LLM đọc danh sách FAQ đã chọn. Nếu FAQ đủ bao phủ toàn bộ câu hỏi, service trả `FaqAnswerResult(answer_markdown, matched_faqs, token_usage)`. Nếu không đủ, trả `None` để pipeline tiếp tục đọc tài liệu bằng PageIndex.
- `/api/faqs/match` là debug endpoint cho FAQ answering, không tăng view count và trả `answer_markdown`, `faq_ids`, `questions`.

11.5. PIPELINE TÍCH HỢP:
- ChatService, ChatStreamService và InquiryService chỉ là adapter API/workflow. Shared core nằm tại `app/modules/rag/query/pipeline.py`.
- Non-stream chat/email gọi `answer_chat` hoặc `answer_email`, nhận `answer_markdown`; rich text HTML conversion nằm ở service layer khi `to_rich_text=True`.
- Stream chat gọi `stream_chat`, forward reasoning/tool/text/final events theo thời gian thực; stream không convert HTML.
- Các source hiện tại: `llm` (PageIndex hoặc direct reply), `faq` (LLM tổng hợp từ FAQ), `bypass` (không có candidate file để đọc).

Quy trình \"Nạp kiến thức\" cực kỳ chi tiết của hệ thống:

BƯỚC 1: UPLOAD & QUICK ACK (`FileService.upload_file_quick`)
- Nhận file qua FastAPI multipart.
- Validate: Max 20MB, đuôi file (.pdf, .md, .docx, .html). Mime type check.
- Metadata check: Bắt buộc có enrollmentYear, academicYear, và type. Các giá trị mặc định được sử dụng nếu thiếu.
- R2 Storage: Đẩy file binary lên storage.
- MongoDB: Tạo record ban đầu status=\"uploading\". Trả về Object thông tin file ngay lập tức.

BƯỚC 2: BACKGROUND PROCESSING (`FileService.process_file_background`)
- Download file từ R2 vào thư mục temp máy chủ.
- LlamaParse: Bóc tách file (vd: PDF, TXT, DOCX) sang Markdown chất lượng cao, giữ được bảng biểu/thời khóa biểu.
- PageIndex:
  - Phân tích MD để tạo cấu trúc mục lục.
  - Sinh ra 1 câu `doc_description` (Mô tả tài liệu).
  - Lưu TOC Tree vào `file_toc_trees` Collection.
- Corpus Indexer:
  - Gắn tài liệu (File) vào các chủ đề tương ứng trên cây Corpus Topic Tree bằng LLM.
  - Lưu liên kết và cập nhật doc_count trên các node chủ đề tương ứng trong MongoDB.
- Artifact storage: Upload bản `.md` đã bóc tách lên R2 (sát cạnh file gốc) để AI tham chiếu nhanh.
- Status Update: Chuyển file sang status="ready" (hoặc "failed" nếu lỗi).

BƯỚC 3: METADATA SYNC (Atomic Counter & MongoDB Index Sync)
- Khi file chuyển sang "ready", hệ thống gọi `$inc: { "total_files": 1 }` cho tất cả nhãn metadata tương ứng trong `metadata_types`.
- Khi user cập nhật metadata qua API `PATCH /api/files/{id}`, hệ thống cập nhật MongoDB và tự động tính toán lại quyền hạn truy cập của các tài liệu trong cây chủ đề.

QUY TẮC ĐỌC FILE:
- Repository active luôn ghép điều kiện `deleted_at=null`; bản ghi đã xóa mềm chỉ xuất hiện qua API trash dành cho admin.
- Admin đọc/list mọi trạng thái file chưa xóa.
- Student và lecture chỉ được list, xem detail, download và đọc TOC của file `status=ready`.
- Student bị ẩn file `lecturer_only=true`; lecture và admin được đọc loại file này.
- List file/trash nhận query `lecturerOnly`; admin/lecture lọc chính xác, còn student không thể dùng `true` để vượt quyền.
- Item của list file/trash là DTO rút gọn, không trả `tableOfContents`; TOC chỉ có trong File detail/TOC API.
- File repository dùng Mongo projection cho list/trash nên không hydrate TOC và các trường ingestion không dùng.
- FAQ list/detail, FAQ-domain retrieval filter và RAG hydration cũng chỉ dùng FAQ có `deleted_at=null`.
- File/FAQ services tự dựng metadata/role filter của domain mình; Corpus chỉ điều phối hai tập payload ID.
- PATCH FAQ chỉ tinh chỉnh document và giữ nguyên Corpus topic assignments; đổi topic qua Corpus API hoặc backfill.

13. HE THONG METADATA TYPES (DEFINITIONS & RULES)
================================================================================

Hệ thống Metadata giúp AI \"lọc\" tài liệu một cách chuẩn xác:

PHÂN LOẠI METADATA (Hardened Schema):
- enrollmentYear: Kiểu YearRange (fromYear, toYear). Mặc định 0-9999. Hỗ trợ nhập một phía (vd: chỉ nhập fromYear thì toYear=9999).
- academicYear: Kiểu YearRange (fromYear, toYear). Tương tự enrollmentYear.
- type: Kiểu mảng (List[DocumentType]). Hỗ trợ lọc nhiều loại tài liệu cùng lúc bằng toán tử $in (Mongo).

QUY TẮC VALIDATION (MetadataService):
- Hệ thống sử dụng một Schema cứng (Hardened Schema) định nghĩa sẵn các trường metadata.
- Mọi API request/response dùng camelCase. DTO Python vẫn khai báo snake_case nội bộ qua `BaseSchema`, alias tự chuyển sang camelCase khi nhận/trả JSON.
- Metadata object-based: Các trường năm học được bọc trong object YearRange để hỗ trợ tìm kiếm khoảng giao thoa.
- Defaults: Nếu không cung cấp metadata khi upload, hệ thống tự động gán giá trị mặc định an toàn (0, 9999, cong_van).
11.2. ATOMIC COUNTER LOGIC (IDEMPOTENCY):
Hệ thống sử dụng cơ chế cập nhật bộ đếm nguyên tử để tránh sai lệch dữ liệu:
- Tăng bộ đếm: Chỉ thực hiện khi lệnh `find_one_and_update` chuyển trạng thái file thành công từ `PROCESSING` -> `READY`.
- Giảm bộ đếm: Chỉ thực hiện khi lệnh `find_one_and_delete` xác nhận file bị xóa đang ở trạng thái `READY`.
- Công nghệ: Sử dụng `$inc` của MongoDB để đảm bảo tính nhất quán ngay cả khi có nhiều request đồng thời.

15. RAG SERVICE (STRATEGY & CORPUS TREE TRAVERSAL FLOW)
================================================================================

`app/modules/rag/query` là shared query package cho chat non-stream, chat stream và email inquiry. Package này chịu trách nhiệm từ câu hỏi đã normalize/analyze đến answer/events, nhưng không chịu trách nhiệm HTTP DTO, SSE JSON formatting, session persistence, rich text HTML hay gRPC commit.

PUBLIC METHODS:
- `answer_chat(input)`: chat non-stream, có chat analyzer/direct reply, trả `RagQueryResult.answer_markdown`.
- `stream_chat(input)`: chat stream, cùng lifecycle với `answer_chat` nhưng yield internal events để ChatStreamService map sang SSE.
- `answer_email(input)`: email inquiry, dùng input đã được `EmailQueryAnalyzerService` chuẩn hóa, không direct reply và dùng chung citation document viewer với chat.

QUY TRÌNH CHUNG:
1.  Analyze:
    - Chat: `ChatQueryAnalyzerService` rewrite câu hỏi, quyết định `needs_rag`, trích xuất metadata filter; nếu thiếu enrollment year thì fallback từ JWT.
    - Email: không chạy chat analyzer; metadata/question đến từ email analyzer/workflow.
2.  Direct reply:
    - Chỉ chat/chat_stream được phép trả lời trực tiếp khi `needs_rag=false`.
    - Direct-answer instruction được truyền bằng Gemini `system_instruction`; user prompt chỉ chứa lịch sử hội thoại gần nhất và tin nhắn hiện tại.
3.  Retrieval:
    - Pre-filter MongoDB theo metadata + user role để có file/FAQ allowed.
    - Corpus Tree Traversal dùng LLM chọn topic/subtree liên quan.
    - Nếu traversal hết `CORPUS_TRAVERSAL_MAX_TURNS` mà chưa select/no-match rõ ràng, pipeline coi là `no_match` và trả no-candidate message thay vì ném lỗi 502.
    - Hydrate supporting FAQ docs trước, rồi Cohere rerank FAQ riêng.
4.  FAQ answering:
    - LLM đọc một hoặc nhiều FAQ đã retrieval/rerank.
    - Nếu FAQ đủ bao phủ toàn bộ câu hỏi, trả `source="faq"` và token usage từ FAQ LLM call.
    - Nếu FAQ chỉ trả lời được một phần, pipeline tiếp tục PageIndex thay vì trả lời thiếu.
    - `BASE_FAQ_ANSWER_SYSTEM_PROMPT` giữ quy tắc full-coverage, metadata và JSON schema; chat dùng `CHAT_FAQ_ANSWER_SYSTEM_PROMPT`, email dùng `EMAIL_FAQ_ANSWER_SYSTEM_PROMPT`.
    - FAQ instruction được truyền bằng Gemini `system_instruction`; user prompt chỉ chứa câu hỏi và danh sách FAQ đã rerank.
5.  PageIndex answering:
    - Chỉ khi FAQ không đủ trả lời toàn bộ câu hỏi, pipeline mới hydrate và Cohere rerank file candidates.
    - Nếu có file candidates, PageIndex agent đọc cấu trúc/nội dung tài liệu bằng tool loop và trả answer Markdown + sources.
    - Nếu không có file candidates và FAQ không đủ, trả no-candidate fallback với `source="bypass"`.

RERANK:
- Integration Cohere nằm ở `app/integrations/cohere/client.py`.
- RAG-specific YAML document builders nằm ở `app/modules/rag/query/retrieval/rerank/cohere_reranker.py`.
- Config chính: `COHERE_API_KEY`, `COHERE_RERANK_ENABLED`, `COHERE_RERANK_MODEL`, `COHERE_RERANK_MAX_CANDIDATES`, `COHERE_RERANK_FILE_TOP_N`, `COHERE_RERANK_FAQ_TOP_N`, `COHERE_RERANK_MAX_TOKENS_PER_DOC`, `COHERE_RERANK_TIMEOUT_SECONDS`.
- `inspect_topic` luôn lấy số sample từ `CORPUS_TRAVERSAL_TOPIC_SAMPLE_LIMIT`; tham số này không được expose cho agent.

16. PAGEINDEX INTEGRATION — AGENTIC TOOLS
================================================================================

PageIndex cung cấp 3 công cụ \"mắt thần\" cho AI Agent, giúp nó chủ động điều hướng tài liệu lớn:

1.  TOOL: `get_document(doc_id: str)`
    - Mô tả: Lấy thông tin chứng minh thư của file.
    - Output: {name, page_count, description}.

2.  TOOL: `get_document_structure(doc_id: str)`
    - Mô tả: Yêu cầu PageIndex trả về mục lục MD phân cấp.
    - Ý nghĩa: AI dùng cái này để biết thông tin tuyển sinh nằm ở Chương 2, dòng 100-200.

3.  TOOL: `get_page_content(doc_id, pages)`
    - Mô tả: Đọc text thô tại các trang/dòng cụ thể.
    - Ý nghĩa: Giúp AI lấy được con số chính xác (vd: 50% học phí) từ văn bản mà không cần nạp context khổng lồ.


17. CHAT SERVICE & AGENTIC RAG LOGIC
================================================================================

Chat layer là adapter mỏng quanh shared `RagQueryPipeline`.

CHAT NON-STREAM:
- `ChatService.generate_chat_response()` load chat history, build `RagQueryInput(mode="chat")`, gọi `answer_chat`.
- Response trả `answer`; nếu `to_rich_text=True`, service convert Markdown sang HTML ở chat layer.
- Interaction logging dùng Markdown gốc, không log HTML.

CHAT STREAM:
- `ChatStreamService.stream_chat_response()` gọi `stream_chat` và map internal events sang JSON SSE hiện tại.
- Stream giữ Markdown/text event realtime, không hỗ trợ rich text conversion.
- Raw Gemini thought/tool output không được persist. Corpus reasoning là một câu ngắn công khai nhưng cũng không lưu MongoDB.

STEPS/SOURCES:
- `query_analysis`: step public có dạng `{type, content}`; `content` hiển thị `effective_question` đã chuẩn hóa nhưng không nhắc riêng `original_question`; dữ liệu phân tích chi tiết còn lại là contract nội bộ của RagQueryPipeline.
- `corpus_tree`: event node-only đã lọc, có `content` mô tả chung, phát một lần cho chat stream và không đưa vào final steps/persistence.
- `corpus_traversal`: step tối giản cho expand/inspect/select/no-match. Expand/inspect dùng `nodeKey`; select dùng `nodeKeys` để chứa toàn bộ topic được chọn. Root topics được inject thẳng vào prompt, không còn tool/list-roots step.
- `faq_retrieval` và `faq_answer`: chỉ xuất hiện khi Corpus traversal trả FAQ candidates; FAQ được chọn và kết quả đủ/chưa đủ để trả lời.
- `file_retrieval`: file candidates sau Cohere rerank; `document_read`: PageIndex đọc cấu trúc hoặc nội dung tài liệu.
- `sources`: citation metadata từ PageIndex tool usage, tiếp tục giữ `originalUrl` và `markdownUrl`; link trong answer luôn dùng `DOCUMENT_VIEW_URL_PREFIX + fileId`.
- `faqRecommendation`: payload typed để FE prefill bản nháp FAQ từ answer đã được admin duyệt. Payload dùng `effectiveQuestion`, metadata FAQ đầy đủ (`0-9999` cho khoảng không xác định) và `lecturerOnly` là phép OR trên matched FAQ cùng file thực sự được PageIndex mở; direct reply trả `null`.
- Candidate file giữ `lecturer_only` ở payload nội bộ để tính recommendation nhưng public `sources` không đổi. Recommendation được persist cùng assistant message và trả lại qua history API.

17.1. SSE (SERVER-SENT EVENTS) PROTOCOL:
- Content-Type: `text/event-stream`.
- Format thông thường: `data: {"type": "text|reasoning|query_analysis|corpus_traversal|faq_retrieval|faq_answer|file_retrieval|document_read", "content": "...", "done": false}`.
- `corpus_tree` mang thêm `tree`; `corpus_traversal` mang thêm `action` và `nodeKey` hoặc `nodeKeys`.
- Final: `data: {"done": true, "sources": [...], "steps": [...], "faqRecommendation": {...}|null, "tokenUsage": {...}, "processingTimeMs": N}`.
- Luồng thường gặp: query_analysis → corpus_tree → reasoning/corpus_traversal × N → faq_retrieval → faq_answer → text → done nếu FAQ đủ; hoặc tiếp tục file_retrieval → document_read × N → text → done.
- CHAT_SYSTEM_PROMPT & EMAIL_SYSTEM_PROMPT dùng base prompt chung, rồi bổ sung khác biệt văn phong/format theo kênh.
- Corpus traversal và document_read activities được stream realtime; non-stream trả toàn bộ timeline trong response cuối.
- [NEW] Error Handling: Khi gặp lỗi 429 (Rate Limit) từ Gemini, hệ thống đẩy chunk lỗi định dạng SSE để Client hiển thị cảnh báo thay vì lỗi 500.

18. AUTH SERVICE & SECURITY LAYER (GRPC)
================================================================================

Dự án tuyệt đối không lưu Password. Mọi bảo mật dựa trên Nest-api gRPC.

CƠ CHẾ GUARD (RequireAuth & RequireAdmin):
1.  AI Service nhận JWT token từ Header.
2.  Gửi gRPC request: `AuthService.VerifyToken(VerifyTokenRequest(token=token))`.
3.  Nest-api thực hiện verify nội bộ. Kết quả trả về `VerifyTokenResponse`.
4.  Payload mapping:
    - `userId` -> sub
    - `user_role` -> role (admin/lecture/student)
    - `access_metadata` -> cohort/department của chính user đó.
5.  Logic:
    - Token lỗi/hết hạn -> 401 Unauthorized.
    - Role != admin (khi gọi endpoint admin) -> 403 Forbidden.
    - Timeout/Service sập -> 503 Service Unavailable.

  - displayName         : str (Tên UI, vd: Năm học)
  - description         : str (Mô tả nhãn)
  - allowedValues       : array [

        "value": str (Mã giá trị),
        "displayName": str (Tên hiển thị),

        "color": str (Mã hex),
        "isActive": bool
      }
    ]
  - totalFiles          : int (Tổng tệp mang tag cha này)
  - isSystem            : bool (Nếu true, cấm xóa nhãn này)
  - isActive            : bool (Soft delete toàn bộ nhãn)

17.4. Collection: `faqs`
  - _id                 : ObjectId
  - question            : str (Câu hỏi gốc có dấu)
  - question_unaccented : str (Câu hỏi không dấu - Dùng cho FTS)
  - answer_rich_text    : str (HTML/Rich text trả lời)
  - answer_markdown     : str (Markdown trả lời)
  - answer_unaccented   : str (Nội dung trả lời không dấu - Dùng cho FTS)
  - metadataFilter      : dict (Dạng {"academicYear": {"fromYear": 2024}, "enrollmentYear": {"fromYear": 2020, "toYear": 2020}})

  - viewCount          : int
  - source              : str (manual, synthesized, seed)

17.5. Collection: `faq_candidates`

21. STORAGE MANAGEMENT (CLOUDFLARE R2 & LOCAL CACHE)
================================================================================

19.1. R2 STORAGE:
- Lưu trữ 2 bucket chính: original-files và markdown-artifacts.
- Mọi truy cập file MD đều qua một lớp Cache logic.

19.2. ARTIFACT CACHING STRATEGY (PageIndex Integration):
Hệ thống sử dụng chiến lược \"Double-Fetch\" để tối ưu hiệu năng và chi phí:
1.  Check Local: Khi cần file MD (cho Agent đọc), hệ thống kiểm tra thư mục `PAGEINDEX_WORKSPACE/artifacts`.
2.  TTL Logic: File cached tại local sẽ tồn tại trong 60 phút (3600s).
3.  Background Invalidation: Một background task kiểm tra và xóa file sau khi quá hạn.
4.  R2 Fallback: Nếu không thấy local cache, hệ thống tải từ R2 về local rồi mới cung cấp cho PageIndex Tool.

Client: r2_storage (app/integrations/storage/client.py).
Giao tiếp thông qua `aioboto3` (Async AWS SDK).

LOGIC QUAN TRỌNG:
- MIME AUTO-REGISTRATION: 
  Trong `FileService`, hệ thống tự động gán Mime Types cho Word/Excel (.docx, .xlsx, .pptx) vì các hệ điều hành Linux tối giản (Docker) thường không nhận diện được các file Office mặc định.
- PATH GENERATION:
  File được lưu theo: `uploads/YYYY/MM/timestamp_original.pdf`. Ngăn chặn việc ghi đè nếu hai người cùng upload file tên \"tailieu.pdf\".

22. RABBITMQ CONSUMER (CONSUMER RESILIENCE)
================================================================================

- Thread Model: Chạy trên `threading.Thread` riêng.
- Retry Loop: 
  Trong hàm `_run()`, nếu connection RabbitMQ bị mất, hệ thống dùng vòng lặp For 1-10 lần, delay tăng dần (`min(5s * attempt, 60s)`) để thử kết nối lại.
- Poison Loop Protection: Nếu parse JSON payload hỏng ngay từ đầu -> Ack message luôn (loại bỏ khỏi hàng chờ) chứ không requeue để tránh kẹt queue mãi mãi.

23. HE THONG EXCEPTION HIERARCHY
================================================================================

Mỗi Exception được map vào một mã lỗi HTTP tương ứng trong Middleware hoặc trả về trực tiếp.

MÃ LỖI & Ý NGHĨA:
- 400 Bad Request (ValidationException): Upload sai định dạng, metadata thiếu.
- 401 Unauthorized: JWT token sai, gRPC Nest-api reject token.
- 403 Forbidden (ForbiddenException): Student cố tình gọi API Admin, Xóa system metadata.
- 404 Not Found (NotFoundException): Không tìm thấy File, Metadata Key.
- 409 Conflict (ConflictException): Upload file trùng tên, Xóa nhãn đang có tài liệu sử dụng.
- 500/502/503 (StorageException, GeminiException, RabbitMQException): Lỗi hạ tầng hoặc API bên thứ ba.

30. PHU LUC: QUY CHUAN XU LY EXCEL & RICH TEXT
================================================================================

Để đảm bảo tính nhất quán giữa các module FAQ và Forms, hệ thống sử dụng một bộ tiện ích chung tại `app/integrations/excel/utils.py`.

CÁC QUY TẮC XỬ LÝ CHÍNH:
1.  Numeric Normalization: 
    - Excel thường lưu số nguyên dưới dạng float (vd: 2020 thành 2020.0).
    - Hệ thống tự động chuyển đổi các giá trị này về chuỗi số nguyên sạch (vd: "2020").
2.  Rich Text Preservation:
    - Sử dụng `openpyxl` để duyệt từng `InlineFont` fragment trong ô Excel.
    - Chuyển đổi các định dạng (Bold, Italic, Underline, Color) sang HTML tương ứng.
3.  Hyperlink Extraction:
    - Tự động trích xuất URL từ các ô có gắn Link.
    - Gộp URL và text hiển thị thành thẻ `<a>`.
4.  Security & UX:
    - Tất cả các thẻ `<a>` được gán `target="_blank"` và `rel="noopener noreferrer"` (xử lý ở tầng Frontend hoặc API Response).
5.  Pydantic Compatibility:
    - Mọi dữ liệu bóc tách từ Excel được ép kiểu về `str` trước khi đưa vào Pydantic model để tránh lỗi `Input should be a valid string`.
