1. TONG QUAN & MUC DICH
================================================================================

AI Service là một FastAPI microservice kết hợp hai chức năng lớn:

  A) EMAIL CLASSIFICATION
     Nhận email từ NestJS qua RabbitMQ, dùng Gemini AI phân loại thành 4 nhãn:
       classRegistration — Yêu cầu đăng ký / hủy / đổi học phần
       inquiry           — Câu hỏi, thắc mắc cần tư vấn
       task              — Nhiệm vụ nội bộ từ phòng ban → phòng giáo vụ
       other             — Không xác định / không liên quan

  B) RAG DOCUMENT SERVICE
     Quản lý kho tài liệu (Gemini File Search + R2) và cung cấp
     chat hỏi đáp dựa trên nội dung tài liệu thực. Bao gồm:
       - Upload / quản lý file (PDF, DOCX, DOC, TXT, MD, HTML, XLSX, XLS, PPTX, CSV, RTF)
       - Quản lý Store (Gemini File Search Store)
       - Quản lý Metadata Type (định nghĩa các trường tag)
       - Chat API: hỏi đáp (query) và streaming (SSE)
       - Tự động soạn email trả lời cho email loại inquiry

Service chạy tại: http://localhost:8000
Swagger UI:       http://localhost:8000/docs

Giao tiếp với:
  - NestJS Backend  → gửi email qua RabbitMQ, nhận kết quả classification
  - Gemini API      → phân loại email + RAG File Search + soạn email
  - MongoDB Atlas   → lưu metadata store / file / metadata_types
  - R2              → lưu file binary (PDF, DOCX...)
  - RabbitMQ        → nhận email từ queue email_ingest_queue

Phiên bản: 3.1.0 (March 2026)
Mục tiêu: Microservice hợp nhất phân loại email và RAG tìm kiếm tài liệu.
Tiếp cận: FastAPI + Google Gemini (google-genai SDK 1.56.0) + MongoDB + R2 Storage.
Hỗ trợ: CamelCase API (Pydantic Aliases) & Snake_case Database Mapping.
Tính năng: Array Metadata Filtering (OR-logic) & Role-based Access Control.

4. KHOI DONG UNG DUNG (LIFESPAN)
================================================================================

Khi chạy `python run.py`, Uvicorn khởi động FastAPI và hàm lifespan()
trong app/main.py thực hiện 5 bước tuần tự:

  BUOC 1: Kết nối MongoDB (Motor async client)
    Database.connect() → tạo pool kết nối (min=10, max=100)
    Nếu thất bại → raise exception, server không khởi động

  BUOC 2: Khởi tạo R2 client (singleton)
    r2_storage → kiểm tra kết nối, list S3 buckets
    Nếu thất bại → chỉ warning (không crash), file upload sẽ lỗi sau

  BUOC 3: Khởi tạo RabbitMQ service
    get_rabbitmq_service() → tạo connection object (lazy connect)
    Nếu thất bại → chỉ warning, RabbitMQ classification không hoạt động

  BUOC 4: Khởi tạo EmailWorkflowOrchestrator
    EmailWorkflowOrchestrator(api_key, model, temperature)
    Tạo: LabelClassifierService + ClassRegistrationService + TaskService + ...
    Nếu thất bại → raise exception (bắt buộc để classification hoạt động)

  BUOC 5: Start RabbitMQ Consumer thread
    Chỉ start nếu cả rabbitmq_service và email_orchestrator đều sẵn sàng
    start_email_ingest_consumer(classifier, loop=event_loop)
    Thread chạy vô tận (daemon=True), lắng nghe queue

5. LUONG XU LY EMAIL (CLASSIFICATION PIPELINE)
================================================================================

Input từ RabbitMQ (NestJS publish):
  {
    "pattern": "ingested",
    "data": {
      "id": 123,
      "subject": "Xin đăng ký môn học",
      "senderEmail": "sv001@example.com",
      "senderName": "Nguyễn Văn A",
      "content": "Em muốn đăng ký..."
    }
  }

Buoc 1: EmailIngestConsumer.on_message()
  - Parse JSON từ bytes → dict
  - Validate với IngestMessage schema (Pydantic)
  - Nếu JSON không hợp lệ → basic_ack (hủy message, tránh poison loop)
  - asyncio.run_coroutine_threadsafe() → gọi orchestrator trong event loop chính
  - Timeout: 120 giây
  - Thành công → basic_ack | Thất bại → basic_nack + requeue=True

Buoc 2: EmailWorkflowOrchestrator.process_request()
  - Gọi LabelClassifierService.classify(title, content) → label
  - Dispatch theo label

Buoc 3: Trả kết quả về NestJS:

  classRegistration:
    { "label": "classRegistration", "internal": {...},
      "extracted": { "studentCode":"SV001", "messageId":123,
                     "items":[{"action":"register","subjectCode":"MATH101",...}] } }

  inquiry:
    { "label": "inquiry", "internal": {...},
      "answer": { "draftSubject":"Re:...", "draftBody":"Kính gửi...",
                  "sources":[...], "messageId":123 } }

  task / other:
    { "label": "task" }  hoặc  { "label": "other" }

6. LABELCLASSIFIERSERVICE - PHAN LOAI EMAIL
================================================================================

File: app/services/orchestration/classification/label_classifier_service.py

CƠ CHE PHAN LOAI:

  Buoc 1: Keyword pre-check (trước khi gọi Gemini — tiết kiệm API)
    Nếu content/title chứa keywords như:
      "đăng ký học phần", "hủy học phần", "xin mở lớp", "mã học phần"...
    → Short-circuit về classRegistration (không gọi Gemini)

  Buoc 2: Gemini AI classification
    Dùng LangChain ChatPromptTemplate + GeminiGenAIChat
    System prompt định nghĩa 4 nhãn + rule ưu tiên:
      classRegistration > task > inquiry > other
    Model trả về 1 token duy nhất (không explanation)

  Buoc 3: Parse & normalize
    Lowercase → kiểm tra VALID_LABELS set
    Aliases: "class_registration" → "classRegistration",
             "question" → "inquiry", "todo" → "task"...
    Fallback: không nhận ra → "other"

4 NHAN CHI TIET:
  classRegistration — sinh viên yêu cầu đăng ký, hủy, đổi, thêm, bật học phần/lớp
  inquiry           — sinh viên hỏi thông tin, xin tư vấn (không yêu cầu hành động)
  task              — yêu cầu từ phòng ban nội bộ giao việc cho phòng Giáo vụ
  other             — spam, không xác định, thiếu ngữ cảnh

chain_prompt(prompt, llm) → LangChain-style chain (prompt | llm)

8. CAC WORKFLOW SERVICE
================================================================================

ClassRegistrationService
  - Gọi Gemini trích xuất học phần đăng ký dạng JSON.
  - Sau đó gọi `GrpcClient.create_class_registration(payload)` gửi thông tin sang nest-api (tạo ticket hỗ trợ).
  - Trả về payload cấu trúc bao gồm `extracted` chứa action (register/cancel/change), mã học phần,...

InquiryService
  1. Trích xuất ý chính (intent/question) và phân loại thắc mắc (inquiry types).
  2. Gọi `draft_inquiry_email_reply()` → Gemini RAG tham khảo tài liệu soạn trả lời (draft).
  3. Nếu messageId != None: gọi `GrpcClient.create_inquiry(...)` gRPC tới nest-api.
     → tạo Gmail draft trên giao diện, đính kèm answer, intent, types và sources (chi tiết tài liệu trích dẫn).
     → Graceful: log warning và bỏ qua nếu nest-api trả UNIMPLEMENTED.
  4. Trả về InquiryResponse (answer, question, types, sources).

TaskService
  - Gọi Gemini trích xuất công việc nội bộ và danh sách người được giao (assignee_ids dạng tên/từ khóa).
  - Gọi gRPC `GrpcClient.find_auth_user_by_keyword(keyword)` đồng thời (asyncio.gather) để tra cứu User IDs thực tế bên nest-api.
  - Tạo task qua `GrpcClient.create_task(payload)`.
  - Trả về label="task" với dữ liệu đã map ID đầy đủ.

OtherService
  Ghi log đối với những email không liên quan, spam... Trả về label="other".

9. EMAIL DRAFT SERVICE - SOAN EMAIL TU DONG
================================================================================

File: app/services/rag/email_draft_service.py

Pure function — KHONG phải HTTP endpoint, gọi nội bộ khi label = inquiry.

  draft_inquiry_email_reply(subject, body, extracted_question, inquiry_types,
                            sender_name, sender_email, additional_context, storeId, metadata_filter) → dict

  Quy trinh:
  1. resolve_store(storeId)
     → storeId có trong request? dùng ngay
     → không? lấy default store từ DB (is_default=True)
     → không có default? log/raise error.

  2. convert_metadata_filter_to_gemini_format(metadata_filter)
     → {"department":"dao_tao"} → 'department="dao_tao"'

  3. Built context string (bao gồm extracted_question và inquiry_types để tập trung LLM suy luận/truy vấn đúng tài liệu).

  4. draft_email_reply(originalSubject, originalBody,
         additionalContext, storeName, metadataFilter)
     → Gemini tìm tài liệu liên quan + soạn email trả lời tự động

  5. Trả về:
     {
       "answer": "Kính gửi em Nguyễn Văn A...",
       "question": "Câu hỏi trích xuất...",
       "types": ["graduation"],
       "filters": {"academicYear": "2024-2025", "cohort": "k20"},
       "sources": [{"title":"...", "text":"...", "url":"..."}],
     }

  Nếu lỗi → log warning, trả về fallback email (tránh crash toàn luồng)

  6. Quá trình tiếp theo do InquiryService điều phối với gRPC (xem Section 8).

10. UPLOAD FILE FLOW
================================================================================

File: app/services/rag/file_service.py

  POST /api/files

  STEP 1: Validation
    validate_file_size()       ← max 20MB (MAX_FILE_SIZE_MB)
    validate_file_extension()  ← pdf, docx, doc, txt, md, html, xlsx, xls, pptx, csv, rtf
    Kiểm tra file trùng tên trong cùng store (ConflictException)

  STEP 2: Validate metadata (BẮT BUỘC)
    accessScope BẮT BUỘC phải có (private, lecture, student)
    Phải có ít nhất academicYear HOẶC cohort (ValidationException nếu thiếu)
    Từng key phải là metadata type đã đăng ký và isActive=True
    Từng value phải nằm trong allowed_values và AllowedValue.isActive=True

  STEP 3: Tạo record MongoDB
    status = UPLOADING

  STEP 4: Upload lên R2
    storage_path = {store_name}/{timestamp}_{filename}
    status → PROCESSING

  STEP 5: Upload Gemini (đồng bộ)
    Upload binary lên Gemini Files API
    Gắn vào File Search store
    Cập nhật MongoDB: geminiDocumentName + status = ACTIVE.
    [Quan trọng]: Lúc này hệ thống tự động gọi `MetadataRepository.sync_metadata_counters(customMetadata, delta=1)`
    (Sử dụng atomic update `$inc`) để tăng biến đếm `totalFiles` cho từng config metadata key tương ứng và value tương ứng bên trong `allowedValues`.
    Nếu lỗi: status = FAILED (file vẫn còn trong R2, có thể retry)
    → Trả về response với status="active"

  FileService Refactoring (High-level coordination):
    - Massive methods decomposed into atomic sub-methods.
    - specialized logic (MIME, ID, Gemini operation waiting) moved to specialized utils.
    - Metadata requirements and role filtration moved to MetadataService.
    - Coordinated retry/rollback using UploadState dataclass.

  CHIẾN LƯỢC XÓA FILE (Explicit Hard Delete):
    - `DELETE /api/files/{fileId}`: Hard delete hoàn toàn. Gỡ file khỏi R2, gỡ Document khỏi Gemini File Search và xóa bản ghi DB. Đồng thời tự động Trigger `MetadataRepository.sync_metadata_counters(delta=-1)` để giảm bộ đếm `totalFiles` trong các metadataTypes.
    - Xóa toàn lô / Batch delete (`DELETE /api/files/all?storeId=...`) cũng lặp qua và áp dụng rollback số lượng `totalFiles` (các record metadata), cũng như xóa triệt để tại (DB + R2 + Gemini).

  Retry file failed:
    POST /api/files/{fileId}/retry
    → File phải ở status FAILED và còn trong R2
    → Gọi lại Gemini upload → status → ACTIVE

11. HE THONG METADATA TYPES
================================================================================

Metadata Type định nghĩa các "trường tag" có thể gắn lên file khi upload.
Dùng để filter documents trong RAG query.

CẤU TRUC MetadataTypeDocument:
  key               : Khóa duy nhất (vd: "academicYear", "accessScope")
  displayName      : Tên hiển thị trên UI (vd: "Năm học")
  description       : Mô tả ngắn
  allowedValues    : List[AllowedValue] hoặc null (null = free text)
  isActive         : bool — false = ẩn, không validate (Soft Delete)
  isSystem         : bool — true = không cho phép xóa qua API
  totalFiles       : int — Biến auto-increment/statistics: Số lượng file hiện đang sử dụng tag này

CẤU TRUC AllowedValue:
  value        : Giá trị trong filter (vd: "student", "2024-2025")
  displayName : Hiển thị UI (vd: "Student", "Năm học 2024-2025")
  isActive    : bool — false = ẩn, không nhận khi upload (Soft Delete)
  color        : Hex color (vd: "#3498DB")
  totalFiles  : int — Số lượng file mang giá trị này
  visibleRoles : array — ["lecture", "student"] hoặc rỗng. 
                 Rỗng = Chỉ Admin thấy. Admin luôn thấy tất cả các giá trị.

DUNG SORT:
  Mặc định trả về kết quả Metadata Type sẽ được ưu tiên `isSystem=True` lên đầu, 
  sau đó mới đến các Metadata Type tùy chỉnh (giảm dần theo thời gian tạo).

CHIẾN LƯỢC XÓA VÀ CẬP NHẬT (UPDATE/DELETE STRATEGIES):
  1. Soft Delete: Thực hiện qua thao tác Update (PUT/PATCH), gán `isActive = False` cho MetadataType hoặc cấu hình AllowedValue.
     Lịch sử và file vẫn giữ tag mà không phá vỡ logic, nhưng UI ngừng đề xuất khi upload file mới.
     - LƯU Ý BẢO VỆ SYSTEM METADATA: 
       + `accessScope`: Bị CẤM thêm value, cập nhật `isActive` cho value, hoặc xóa value.
       + `academicYear` & `cohort`: Bị CẤM cập nhật `isActive` hoặc xóa đối với value `all`.
       + Bị CẤM update `isActive` cho bản thân các System Metadata Type.
  2. Hard Delete: 
     - `DELETE /api/metadata/{key}` hoặc `DELETE /api/metadata/{key}/values/{value}`
     - API Explicit Hard Delete có cơ chế bảo vệ: CHỈ cho phép xóa khi `total_files == 0`.
     - Nếu `total_files > 0`, trả về 409 Conflict.
     - System metadata (`isSystem = True`) và các value được bảo vệ sẽ bị cấm xóa.

3 SYSTEM METADATA (seed bởi scripts/seed_metadata.py, isSystem=True):

  academicYear (Năm học):
    Chứa entry value="all" trong allowed_values.
    Giá trị: 2020-2021, 2021-2022, 2022-2023, 2023-2024, 2024-2025, 2025-2026, all

  cohort (Khóa sinh viên):
    Chứa entry value="all" trong allowed_values.
    Giá trị: K62, K63, K64, K65, K66, K67, K68, all

  accessScope (Phạm vi truy cập):
    has_all_value = False
    Giá trị: private (Admin Only), lecture (Lecture Only), student (Student Only)

QUY TAC VALIDATE khi upload file:
  1. accessScope BAT BUOC phải có và là định dạng Array List (chỉ chấp nhận các giá trị: ["private"], ["student"], ["lecture"]).
  2. Ràng buộc độc quyền: File CHỈ ĐƯỢC MANG 1 tính chất truy cập duy nhất. Nếu chứa `private` thì KHÔNG ĐƯỢC kèm theo `student` hay `lecture`. Ngược lại tính public (`student`/`lecture`) KHÔNG ĐƯỢC CHỨA `private`.
  3. Phải có ít nhất academicYear HOAC cohort
  4. Mỗi key phải là metadata type đã đăng ký và isActive=True
  5. Mỗi value phải nằm trong allowed_values và AllowedValue.isActive=True

  Vi dụ customMetadata hợp lệ:
    {"academicYear":"2024-2025","cohort":"K65","accessScope":"student"}

12. FILTER BUILDER & ROLE-BASED ACCESS
================================================================================

File: app/services/rag/utils/filter_builder.py

convert_metadata_filter_to_gemini_format(metadata: dict) → str|None

  Vi dụ đơn giản:
    Tất cả request filter (khi GET files hoặc Search) BẮT BUỘC phải truyền value dưới định dạng list (mảng).
    Input:  {"academicYear": ["2024-2025"], "department": ["dao_tao"]}
    Output: 'academicYear="2024-2025" AND department="dao_tao"'

  Vi dụ list (OR condition - cùng key):
    Input:  {"cohort": ["K65", "K66"]}
    Output: '(cohort="K65" OR cohort="K66")'

  Vi dụ multiple keys (AND condition - khác key):
    Input:  {"academicYear": ["2024"], "cohort": ["K65"]}
    Output: 'academicYear="2024" AND cohort="K65"'

  Vi dụ support_all_value=True (academicYear):
    Input:  {"academicYear": ["2024-2025"]}
    Output: '(academicYear="2024-2025" OR academicYear="all")'
    → has_all_value=True khi allowed_values chứa entry với value="all"

  Input null / empty → Trả về None (không filter Gemini)

Phan quyen theo role trong Chat API:
  role="student"  → Thêm filter accessScope IN ["student"]
  role="lecture"  → Thêm filter accessScope IN ["lecture"]
  role="admin"    → Không giới hạn accessScope (có thể xem tất cả gồm "private")
  role="admin"    → Admin luôn bypass filter visibleRoles và có thể xem toàn bộ metadata.

20. RABBITMQ CONSUMER

File: app/services/messaging/email_ingest_consumer.py

Chạy trong thread riêng biệt (không trong FastAPI event loop).

start_email_ingest_consumer(classifier, loop) → threading.Thread
  - declare_email_ingest_queue() → đảm bảo queue tồn tại (durable=True)
  - channel.basic_consume(queue, on_message, auto_ack=False)
  - Chạy channel.start_consuming() trong thread daemon

on_message(ch, method, properties, body):
  1. _safe_json_loads(body) → parse bytes → dict
     Lỗi JSON → basic_ack (hủy message)
  2. IngestMessage.model_validate(payload)
     Lỗi schema → basic_ack (tránh poison loop)
  3. asyncio.run_coroutine_threadsafe(classifier.process_request(...), loop)
     Kết nối thread RabbitMQ với asyncio event loop
  4. fut.result(timeout=120)
     Thành công → basic_ack
     Timeout / exception → basic_nack + requeue=True

Retry ket noi RabbitMQ:
  Tối đa 10 lần, delay bắt đầu từ 5s, tăng dần, tối đa 60s
  Exponential backoff

RabbitMQ Service (rabbitmq_service.py):
  get_rabbitmq_service() → singleton
  declare_email_ingest_queue() → idempotent, durable=True
  publish(queue, message) → gửi message dạng JSON
