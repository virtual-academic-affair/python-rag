1. PHẠM VI HỆ THỐNG
================================================================================

AI Service là FastAPI service cho bốn nhóm nghiệp vụ:

1. Email automation
   - Nhận email ingest từ RabbitMQ.
   - Kiểm tra state của message qua gRPC khi GRPC_ENABLED=true.
   - Phân loại và chạy class-registration hoặc inquiry workflow.
   - Gửi trạng thái xử lý qua WebSocket.

2. Knowledge ingestion
   - Quản lý FileDocument và file gốc trên R2.
   - LlamaParse chuyển tài liệu sang Markdown.
   - PageIndex tạo TOC, description và line count.
   - CorpusLinker gán file/FAQ vào Corpus Topic Tree.

3. Shared RAG query
   - Dùng chung cho chat non-stream, chat stream và email inquiry.
   - FAQ-first; chỉ đọc file bằng PageIndex khi FAQ chưa đủ trả lời.
   - Không dùng vector database hoặc embedding search.

4. Catalog/admin
   - File, FAQ, Forms, Metadata schema và Corpus Topic Tree APIs.
   - Soft-delete/restore/purge cho File và FAQ.

FAQ auto-synthesis, FAQ review candidate và interaction logging đã bị loại khỏi
runtime/API. Corpus contract `FaqCandidate` vẫn tồn tại vì đây là retrieval seed,
không phải synthesis candidate.

3. STARTUP VÀ SHUTDOWN
================================================================================

FastAPI lifespan trong app/main.py thực hiện:

Startup:
1. Kết nối MongoDB và init Beanie cho:
   - FileDocument, FileTocTree, FaqDocument;
   - ChatSessionDocument, ChatMessageDocument;
   - FormDocument, CorpusNodeDocument.
2. Probe R2 bằng list_buckets; lỗi chỉ warning.
3. Khởi tạo RabbitMQ wrapper nếu RABBITMQ_ENABLED=true.
4. Tạo EmailWorkflowOrchestrator và lưu vào app.state.
5. Khởi động email consumer trong daemon thread nếu RabbitMQ sẵn sàng.
6. Khởi động asyncio task dọn PageIndex Markdown local mỗi 30 phút.
7. Kết nối Redis nếu REDIS_ENABLED=true; lỗi chỉ warning.

Shutdown:
1. Đóng RabbitMQ connection.
2. Ngắt MongoDB.
3. Cancel và await artifact-cleanup task.
4. Đóng HTTP clients do LiteLLM tạo khi event loop còn sống.
5. Ngắt Redis.

Các bước close/await cuối cùng ngăn aiohttp session và Queue.get task bị bỏ dở khi
process tắt.


4. AUTH VÀ ROLE
================================================================================

REST authentication không gọi gRPC. app/core/auth.py verify JWT cục bộ:

- Algorithm: HS256.
- Secret/audience/issuer: JWT_SECRET, JWT_TOKEN_AUDIENCE, JWT_TOKEN_ISSUER.
- Header: Authorization: Bearer <token>.
- sub được chuẩn hoá thành string.

Role contract: student | lecture | admin.

- require_auth: bất kỳ JWT hợp lệ.
- require_admin: role phải đúng bằng admin.
- lecture là privileged cho visibility, nhưng không thay thế admin ở mutation API.
- Student không thấy entity lecturerOnly=true; detail API trả 404 thay vì làm lộ
  sự tồn tại của entity.

WebSocket xác thực bằng message đầu tiên sau khi accept:

- /api/files/progress/{clientId}: JWT hợp lệ.
- /api/email/progress/{clientId}: JWT admin.
- Server trả {"type":"auth_ok"} khi thành công.


5. FILE INGESTION VÀ LIFECYCLE
================================================================================

Upload nhanh:
1. POST /api/files hoặc /api/files/batch validate size/extension/metadata.
2. Tạo FileDocument status=uploading.
3. Upload file gốc lên R2.
4. Trả response 201 và xếp background task xử lý tiếp.

Background ingestion:
1. Chuyển status sang processing và phát progress events.
2. LlamaParse tạo Markdown.
3. Upload Markdown lên R2.
4. PageIndex tạo structure, description, lineCount.
5. Upsert FileTocTree với markdownStoragePath canonical trên R2.
6. Warm pageindex:doc:{fileId} sau khi R2 upload và TOC upsert thành công.
7. CorpusLinker dùng tên/description/TOC để gán topic.
8. Cập nhật FileDocument status=ready và metadata retrieval cần thiết.

Nếu background ingestion lỗi, file chuyển failed và cleanup các artifact đã tạo
theo best-effort.

Soft delete:
- Lưu deletedAt/deletedBy và topic assignments cũ.
- Gỡ Corpus links, evict PageIndex/local Markdown và RAG entity cache.
- Giữ R2/TOC để có thể restore.

Restore:
- Kiểm tra file gốc; READY file còn cần Markdown và TOC.
- Ưu tiên topic assignments cũ còn tồn tại; nếu không còn thì chạy CorpusLinker.

Purge:
- Chỉ cho file đã soft-delete.
- Gỡ Corpus, evict PageIndex, xóa TOC, xóa original/Markdown R2 rồi xóa Mongo doc.


6. PAGEINDEX METADATA VÀ MARKDOWN CACHE
================================================================================

Redis key: pageindex:doc:{docId}

Payload hợp lệ cần có:
- structure;
- doc_name, doc_description, line_count;
- markdown_storage_path.

_load_document_metadata(docId):
1. Thử Redis.
2. Payload cũ/thiếu markdown_storage_path là miss.
3. Miss thì đọc FileTocTree từ MongoDB.
4. Gắn canonical local path PAGEINDEX_WORKSPACE/{docId}.md.
5. Ghi metadata hoàn chỉnh về Redis.
6. Không đọc filesystem và không tải R2.

Caller:
- get_document và get_document_structure chỉ tải metadata/structure.
- get_page_content tải metadata rồi gọi Markdown loader.

Markdown loader:
- Lock theo docId trong process.
- File local còn TTL được touch và tái sử dụng.
- Missing/expired thì download markdown_storage_path từ R2, ghi temp và atomic replace.
- R2 lỗi raise exception; không trả content rỗng.
- Cleanup task xóa local artifact hết TTL.


7. CORPUS TOPIC TREE
================================================================================

CorpusNodeDocument lưu:
- nodeKey, title, summary, parentKey, childKeys;
- directFileIds/directFaqIds;
- subtreeFileIds/subtreeFaqIds và counts suy ra.

Admin APIs quản lý create/update/delete/merge và manual payload assignments.
Repository rebuild node/ancestors sau mutation và kiểm tra cycle/integrity.

Backfill:
- Chạy trong background job có guard in-process chống chạy đồng thời.
- Reset links, index từng READY file chưa xóa và từng FAQ chưa xóa.
- Khi tree rỗng, CorpusLinker có thể đề xuất root topics mới.

Traversal:
1. Build typed cached Corpus snapshot.
2. Prefilter allowed file/FAQ IDs theo metadata và role.
3. LLM duyệt/inspect topic tree với giới hạn cấu hình.
4. Trả typed FileCandidate và Corpus FaqCandidate seeds theo thứ tự chọn.


8. SHARED RAG QUERY PIPELINE
================================================================================

app/modules/rag/query/pipeline.py phục vụ:
- answer_chat;
- stream_chat;
- answer_email.

Luồng non-stream:
1. Analyze câu hỏi.
   - Chat: rewrite, needsRag gate, metadata filter và enrollment fallback.
   - Email: normalize subject/body, inquiryTypes và metadata filter; luôn chạy RAG.
2. Nếu chat không cần retrieval, generate direct reply và trả source=llm:
   - greeting/thanks được phản hồi ngắn gọn;
   - câu hỏi ngoài phạm vi Giáo vụ bị từ chối, không trả lời kiến thức tổng quát.
3. Traverse Corpus Tree để lấy FAQ/file seeds.
4. Hydrate và Cohere-rerank FAQ.
5. FAQ answering LLM đọc một hoặc nhiều FAQ.
6. Nếu đủ bao phủ: trả source=faq, không hydrate/rerank file.
7. Nếu chưa đủ: hydrate và Cohere-rerank file.
8. Không có file: trả no-candidate message, source=bypass.
9. Có file: PageIndex agent dùng get_document_structure/get_page_content, trả answer,
   steps, sources, tokenUsage và maxTurnsReached.

Chat stream dùng cùng semantic flow nhưng phát SSE events trong lúc traversal và
PageIndex agent chạy. Adapter chat chịu trách nhiệm JSON/SSE, session persistence và
faqRecommendation; email adapter chịu trách nhiệm rich text và gRPC side effects.


9. FAQ-FIRST ANSWERING
================================================================================

FAQ module hiện chỉ gồm:
- CRUD, list/search;
- bulk JSON;
- Excel/CSV import preview và apply;
- soft-delete/restore/purge;
- admin debug POST /api/faqs/match.

FaqDocument fields public chính:
- faqId, question, answerRichText;
- metadataFilter, lecturerOnly;
- viewCount, source;
- createdAt, updatedAt, deletedAt, deletedBy.

source mới: manual | bulk_import | seed. Giá trị synthesized có thể còn trên dữ
liệu lịch sử và được giữ nguyên như provenance.

Không còn candidateId, faq_candidates, interaction_logs, synthesis scheduler hoặc
synthesis/candidate endpoints. faqRecommendation trong chat chỉ là DTO prefill và
không tự tạo FAQ.


10. RERANK VÀ ENTITY HYDRATION
================================================================================

RetrievalService xử lý FAQ và file riêng:

- FAQ: hydrate -> Cohere rerank -> COHERE_RERANK_FAQ_TOP_N.
- File: chỉ hydrate khi FAQ chưa trả lời -> rerank -> COHERE_RERANK_FILE_TOP_N.
- Nếu Cohere bị tắt/thiếu key/lỗi, giữ thứ tự Corpus traversal.
- Candidate pool lớn hơn COHERE_RERANK_MAX_CANDIDATES bị từ chối bởi contract.

File entity cache gồm retrieval/source fields từ FileDocument + FileTocTree.
FAQ entity cache gồm question, answer, metadata và visibility cần cho answering.
MGET miss chỉ batch-query MongoDB cho IDs miss và giữ candidate order.
Presigned/public URL không nằm trong entity cache.


11. REDIS CACHE CONTRACT
================================================================================

Revision keys không TTL:
- rag:revision:corpus
- rag:revision:file_eligibility
- rag:revision:faq_eligibility

Payload keys:
- rag:corpus:{corpusRevision}                         TTL 300s
- rag:allowed:file:{revision}:{scope}:{metadataHash} TTL 120s
- rag:allowed:faq:{revision}:{scope}:{metadataHash}  TTL 120s
- rag:file:{fileId}                                  TTL 600s
- rag:faq:{faqId}                                    TTL 600s
- pageindex:doc:{docId}                              TTL 3600s

scope chỉ là student hoặc privileged. Metadata hash dùng canonical sorted JSON.

Invalidation:
- File mutation: delete rag:file:{fileId}; eligibility mutation tăng file revision.
- FAQ mutation: delete rag:faq:{faqId}; eligibility mutation tăng FAQ revision.
- Corpus mutation/reindex/unindex: tăng corpus revision.
- TOC mutation: invalidate file entity và warm/evict PageIndex metadata.

Redis client/cache là best-effort. Query fallback MongoDB; mutation chính vẫn thành
công nếu INCR/DELETE cache thất bại và warning được ghi log.


12. CHAT VÀ SESSION PERSISTENCE
================================================================================

POST /api/chat/query:
- Nhận question, optional sessionId và toRichText.
- Tạo session nếu thiếu ID, đọc tối đa 6 history messages, persist user message.
- Persist assistant answer cùng sources/steps/tokenUsage/faqRecommendation.

POST /api/chat/stream:
- Nhận question và optional sessionId.
- SSE `data: <JSON>\n\n`.
- Event đầu tiên có sessionId.
- Có events query_analysis, corpus_tree, corpus_traversal,
  corpus_traversal_end, FAQ/file pipeline steps, PageIndex call/text.
- Event cuối có done=true, source, sources, steps, tokenUsage,
  processingTimeMs và faqRecommendation.
- Không hỗ trợ toRichText ở stream.

Session APIs chỉ cho owner của JWT: list, messages, rename, archive, unarchive, delete.


13. EMAIL INGEST VÀ WORKFLOWS
================================================================================

RabbitMQ consumer chạy trong daemon thread, chuyển coroutine sang main event loop
bằng asyncio.run_coroutine_threadsafe và chờ tối đa 300 giây.

Message flow:
1. Invalid JSON/schema: ACK để tránh poison loop.
2. Nếu gRPC bật, GetState phải cho biết message current và chưa có records.
3. Phát email_processing_started qua WebSocket channels.
4. EmailWorkflowOrchestrator phân loại và chạy workflow.
5. Thành công: phát done và ACK.
6. Retryable/commit error: republish với retry header đến
   RABBITMQ_EMAIL_MAX_RETRIES; hết retry thì NACK requeue=false vào DLQ.
7. Permanent/unexpected error: DLQ.

InquiryService gọi RagQueryPipeline.answer_email. FAQ/file retrieval giống chat;
citation file được resolve bằng DOCUMENT_VIEW_URL_PREFIX + fileId. gRPC create/commit
vẫn nằm trong email workflow, không nằm trong RAG pipeline.


14. METADATA
================================================================================

Year range public:
{
  "fromYear": 0,
  "toYear": 9999
}

0 và 9999 là sentinel mở. fromYear phải <= toYear.

File metadata:
- enrollmentYear;
- academicYear;
- type: ctdt | cong_van | quyet_dinh.

FAQ metadata chỉ gồm enrollmentYear và academicYear.
Unified retrieval filter có thể thêm type array.

GET /api/metadata/schema trả documentTypes, yearMin, yearMax và không đọc DB.


15. FORMS
================================================================================

Forms lưu documentType, contentLink rich text và optional notes.
API hỗ trợ list/search, CRUD, Excel/CSV import preview và apply.
Excel helpers chuẩn hóa numeric cell, giữ rich text/hyperlink khi parser hỗ trợ và
thêm target/rel an toàn cho link ở output workflow tương ứng.
