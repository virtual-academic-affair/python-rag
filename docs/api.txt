XÁC THỰC (AUTHENTICATION)
════════════════════════════════════════════════════════════════════════════════

Các API quản lý (Store, File, Metadata write) yêu cầu JWT token với role admin.
Token được xác thực qua gRPC call tới nest-api (AuthService.VerifyToken).

Header bắt buộc:
  Authorization: Bearer <jwt_token>

Token lấy từ nest-api (POST /auth/login hoặc GET /auth/google).
Role phải là "admin" để gọi các API được bảo vệ.

LỖI AUTH:
  401 Unauthorized — Không có hoặc sai format Authorization header
  403 Forbidden    — Token hợp lệ nhưng role không phải admin
  503 Service Unavailable — Không thể kết nối tới nest-api gRPC

                       2. EMAIL CLASSIFICATION APIs
════════════════════════════════════════════════════════════════════════════════

Service nhận email (qua RabbitMQ hoặc HTTP), phân loại và trả về label + payload.
Có 4 label: classRegistration | task | inquiry | other

────────────────────────────────────────────────────────────────────────────────
2.1. POST /process - Phân loại email thủ công
────────────────────────────────────────────────────────────────────────────────

MÔ TẢ:
  Phân loại email từ HTTP request (không qua RabbitMQ).
  Được NestJS gọi trực tiếp khi cần phân loại thủ công.
 
LUỒNG XỬ LÝ (WORKFLOW):
  1. Nhận title/content từ request body.
  2. Gọi EmailWorkflowOrchestrator để điều phối:
     a. LabelClassifierService kiểm tra keyword pre-check (tiết kiệm API).
     b. Nếu không match, gọi Gemini AI phân loại theo 4 nhãn.
     c. Dispatch tới service nghiệp vụ (ClassRegistration/Inquiry/Task/Other).
     d. Nếu là 'inquiry', thực hiện RAG tìm kiếm tài liệu để soạn draft reply.
  3. Trả về label đã phân loại và dữ liệu trích xuất/soạn thảo.

REQUEST EXAMPLE:
  {
    "title": "Xin đăng ký môn học",
    "content": "Em muốn đăng ký môn Toán cao cấp, mã môn MATH101..."
  }

RESPONSE 200 OK (label = classRegistration):
  {
    "messageId": 123,
    "label": "classRegistration",
    "extracted": {
      "studentCode": "SV001",
      "studentName": "Nguyễn Văn A",
      "academicYear": 2024,
      "note": "",
      "messageId": 123,
      "items": [
        {
          "action": "register",
          "subjectName": "Toán cao cấp",
          "subjectCode": "MATH101",
          "className": "L01",
          "slotInfo": "",
          "isInCurriculum": true
        }
      ]
    }
  }

RESPONSE 200 OK (label = inquiry):
  {
    "messageId": 123,
    "label": "inquiry",
    "inquiry": {
      "answer": "Kính gửi...",
      "question": "Câu hỏi của bạn",
      "types": ["graduation"],
      "filters": {
        "academicYear": ["2024-2025"],
        "cohort": ["k20"]
      },
      "sources": [
        {
          "citationId": 1,
          "title": "Quy chế đào tạo 2026",
          "text": "Đoạn văn liên quan...",
          "url": "http://r2.../presigned-url",
          "fileId": "67a5..."
        }
      ],
      "messageId": 123
    }
  }

RESPONSE 200 OK (label = task | other):
  {
    "messageId": 123,
    "label": "task"
  }

WORKFLOW:
  1. Nhận request với internal + title + content
  2. LLM (Gemini) phân loại → classRegistration / task / inquiry / other
  3. classRegistration: Extract thêm chi tiết đăng ký môn học
  4. inquiry: 
     a. Trích xuất metadata (Rule-based: Regex) cho `academicYear` và `cohort`.
     b. Tự động hỗ trợ Tiếng Việt không dấu và các ký tự viết tắt (NH, NK, K).
     c. Gọi RAG service để draft email reply (dùng default store).
     d. **OR all logic**: Tự động mở rộng filter sang giá trị "all" nếu trích xuất được giá trị cụ thể.
  5. Trả về response tương ứng với label

────────────────────────────────────────────────────────────────────────────────
2.2. POST /api/test/classification/ingested - Test từ RabbitMQ payload
────────────────────────────────────────────────────────────────────────────────

MÔ TẢ:
  Test endpoint: mô phỏng xử lý một message từ RabbitMQ (pattern "ingested").
  Dùng để debug pipeline mà không cần kết nối RabbitMQ thực.

REQUEST EXAMPLE:
  {
    "pattern": "ingested",
    "data": {
      "messageId": 123,
      "subject": "Xin đăng ký môn học",
      "senderEmail": "student@example.com",

    }
  }

RESPONSE 200 OK:
  {
    "success": true,

════════════════════════════════════════════════════════════════════════════════
                        3. STORE MANAGEMENT APIs
════════════════════════════════════════════════════════════════════════════════

🔒 XÁC THỰC BẮT BUỘC: Tất cả endpoint trong section này yêu cầu:
   Header: Authorization: Bearer <jwt_token>  (role phải là "admin")
Store là File Search store của Gemini API, dùng để lưu trữ documents cho RAG.
Mỗi store có thể chứa nhiều documents và được sử dụng cho Chat / Email queries.

────────────────────────────────────────────────────────────────────────────────
3.1. POST /api/stores - Tạo Store mới
────────────────────────────────────────────────────────────────────────────────

MÔ TẢ:
  Tạo một File Search store mới trên Gemini API và MongoDB.

REQUEST EXAMPLE:
  {
    "displayName": "Văn bản Đào tạo 2026",  // required, 1-512 chars
    "setAsDefault": true                     // optional, default: false
  }

RESPONSE 201 Created:
  {
    "storeId": "67a5f2c8abc123def456789",
    "storeName": "fileSearchStores/van-ban-dao-tao-2026-abc123",
    "displayName": "Văn bản Đào tạo 2026",
    "fileCount": 0,
    "totalSize": 0,
    "isDefault": true,
    "createdAt": "2026-03-11T10:30:00.000000",
    "updatedAt": "2026-03-11T10:30:00.000000"
  }

RESPONSE 409 Conflict:
  { "detail": "Store name 'Văn bản Đào tạo 2026' already exists" }

WORKFLOW:
  1. Validate displayName
  2. Check duplicate trong MongoDB
  3. Generate store_name từ displayName + random suffix
  4. Gọi Gemini API tạo file search store
  5. Nếu set_as_default=true: unset các store khác, set store này là default.
  6. Lưu thông tin vào MongoDB (collection file_search_stores).
  7. Trả về thông tin store vừa tạo.

────────────────────────────────────────────────────────────────────────────────
3.2. GET /api/stores - Liệt kê Stores
────────────────────────────────────────────────────────────────────────────────

MÔ TẢ:
  Liệt kê tất cả stores với phân trang.
  Dùng is_default=true để lấy default store.

QUERY PARAMETERS:
  - isDefault: bool (optional) - Lọc theo default store
  - page: int (optional, default: 1)
  - limit: int (optional, default: 50, max: 100)

REQUEST EXAMPLE:
  GET /api/stores?isDefault=true
  GET /api/stores?page=1&limit=10

RESPONSE 200 OK:
  {
    "stores": [
      {
        "storeId": "67a5f2c8abc123def456789",
        "storeName": "fileSearchStores/van-ban-dao-tao-2026-abc123",
        "displayName": "Văn bản Đào tạo 2026",
        "fileCount": 42,
        "totalSize": 52428800,
        "isDefault": true,
        "createdAt": "2026-03-11T10:30:00.000000",
        "updatedAt": "2026-03-11T10:30:00.000000"
      }
    ],
    "total": 3,
    "page": 1,
    "limit": 50
  }
 
LUỒNG XỬ LÝ (WORKFLOW):
  1. Guard: require_auth.
  2. Nếu query isDefault=true, tìm store có is_default: true trong MongoDB.
  3. Ngược lại, thực hiện truy vấn phân trang (skip/limit) trong DB.
  4. Trả về danh sách kết quả.


────────────────────────────────────────────────────────────────────────────────
3.4. GET /api/stores/{storeId} - Lấy chi tiết Store
────────────────────────────────────────────────────────────────────────────────

MÔ TẢ:
  Lấy thông tin chi tiết của một store theo MongoDB ObjectId.

RESPONSE 200 OK: (cùng format StoreDetailResponse)

RESPONSE 404 Not Found:
  { "detail": "Store 67a5... not found" }
 
LUỒNG XỬ LÝ (WORKFLOW):
  1. Guard: require_auth.
  2. Truy vấn MongoDB theo storeId.
  3. Trả về chi tiết store hoặc lỗi 404.

────────────────────────────────────────────────────────────────────────────────
3.5. PATCH /api/stores/{storeId} - Cập nhật Store
────────────────────────────────────────────────────────────────────────────────

MÔ TẢ:
  Cập nhật displayName hoặc set làm default store.

REQUEST EXAMPLE:
  {
    "displayName": "Tên mới",   // optional
    "isDefault": true           // optional
  }

RESPONSE 200 OK: (StoreDetailResponse đã cập nhật)

RESPONSE 404 Not Found:
  { "detail": "Store not found" }
 
LUỒNG XỬ LÝ (WORKFLOW):
  1. Guard: require_auth.
  2. Cập nhật displayName/isDefault trong MongoDB.
  3. Nếu set as default, tự động cập nhật lại các store khác trong DB.
  4. Trả về dữ liệu sau khi update.

────────────────────────────────────────────────────────────────────────────────
3.6. DELETE /api/stores/{storeId} - Xóa Store
────────────────────────────────────────────────────────────────────────────────

MÔ TẢ:
  Xóa một store (full delete: Gemini + R2 + MongoDB).

QUERY PARAMETERS:
  - force: bool (optional, default: false)
    + false: Từ chối nếu store còn files
    + true: Xóa store cùng toàn bộ files

RESPONSE 204 No Content

RESPONSE 400 Bad Request:
  { "detail": "Store still has files. Use force=true to delete anyway." }

RESPONSE 404 Not Found:
  { "detail": "Store not found" }
 
LUỒNG XỬ LÝ (WORKFLOW):
  1. Guard: require_auth.
  2. Nếu force=false: Kiểm tra store có chứa file không. Nếu có, báo lỗi 400.
  3. Nếu force=true: Thực hiện xóa hết file trong store (Gemini + R2 + DB) trước.
  4. Gọi Gemini API xóa store trên cloud.
  5. Xóa bản ghi trong MongoDB.

────────────────────────────────────────────────────────────────────────────────
3.7. DELETE /api/stores/all - Xóa tất cả Stores
────────────────────────────────────────────────────────────────────────────────

MÔ TẢ:
  Xóa toàn bộ stores (full delete: Gemini + R2 + MongoDB).
  ⚠️ NGUY HIỂM: Không thể hoàn tác!

RESPONSE 200 OK:
  {
    "deletedCount": 3,
    "message": "Deleted 3 stores (full delete)"
  }
 
LUỒNG XỬ LÝ (WORKFLOW):
  1. Duyệt danh sách tất cả store trong MongoDB.
  2. Thực hiện xóa triệt để từng store (Gemini API + R2 files + DB records).
  3. Trả về tổng số lượng store đã xóa thành công.


────────────────────────────────────────────────────────────────────────────────
3.9. POST /api/stores/{storeId}/sync - Đồng bộ thống kê Store
────────────────────────────────────────────────────────────────────────────────

MÔ TẢ:
  Cập nhật file_count và total_size từ Gemini API vào MongoDB.

RESPONSE 200 OK: (StoreDetailResponse với thống kê đã cập nhật)

RESPONSE 404 Not Found:
  { "detail": "Store not found" }
 
LUỒNG XỬ LÝ (WORKFLOW):
  1. Guard: require_auth.
  2. Gọi Gemini API lấy thông số thực tế (active_documents_count, size_bytes).
  3. Cập nhật file_count và total_size tương ứng vào MongoDB.

                        4. METADATA TYPE APIs
════════════════════════════════════════════════════════════════════════════════

🔒 XÁC THỰC:
   - GET /api/metadata                        → Authentication: Required (any valid user)
   - GET /api/metadata/{key}                  → Authentication: Required (any valid user)
   - POST /api/metadata                       → Yêu cầu role admin
   - PATCH /api/metadata/{key}                → Yêu cầu role admin
   - POST /api/metadata/{key}/values          → Yêu cầu role admin
   - PATCH /api/metadata/{key}/values/{value} → Yêu cầu role admin
   - DELETE /api/metadata/{key}               → Yêu cầu role admin
   - DELETE /api/metadata/{key}/values/{value}→ Yêu cầu role admin

Metadata Type là định nghĩa các trường metadata có thể gắn vào file khi upload.
Được dùng để filter documents trong RAG (Gemini File Search filter string).

GHI CHÚ VỀ CHUẨN HÓA (CASE NORMALIZATION):
  - Hệ thống tự động chuẩn hóa metadata keys và list string values về snake_case trong DB.
  - API hỗ trợ nhận cả camelCase và snake_case cho key và value (input flexibility).
  - Kết quả trả về từ API sẽ duy trì camelCase cho keys để đồng bộ project.

────────────────────────────────────────────────────────────────────────────────
4.1. POST /api/metadata - Tạo Metadata Type
────────────────────────────────────────────────────────────────────────────────

MÔ TẢ:
  Tạo một metadata field type mới.

REQUEST EXAMPLE:
  {
    "key": "department",              // required, unique
    "displayName": "Phòng ban",       // required
    "description": "Phòng ban phụ trách tài liệu",
    "allowedValues": [
      {
        "value": "dao_tao",
        "displayName": "Đào tạo",
        "isActive": true,
        "color": "#3498DB",
        "visibleRoles": ["lecture", "student"]
      },
      {
        "value": "khcn",
        "displayName": "KHCN",
        "isActive": true,
        "color": "#2ECC71",
        "visibleRoles": ["lecture", "student"]
      },
      {
        "value": "all",
        "displayName": "Tất cả",
        "isActive": true,
        "color": "#95A5A6",
        "visibleRoles": ["admin", "lecture", "student"]
      }
    ]
  }

GHI CHÚ:
  - hasAllValue tự động tính từ allowedValues: true nếu allowedValues chứa value="all"
  - Để filter builder tự thêm OR key="all", hãy thêm AllowedValue với value="all" vào danh sách

GHI CHÚ:
  - isActive mặc định là true (không cần truyền khi tạo)
  - isSystem mặc định là false (không cần truyền khi tạo)
  - color chỉ có ở allowedValues, không có ở metadata type level

RESPONSE 201 Created:
  {
    "metadataId": "67a5f2c8abc123def456790",
    "key": "department",
    "displayName": "Phòng ban",
    "description": "Phòng ban phụ trách tài liệu",
    "allowedValues": [
      {
        "value": "dao_tao",
        "displayName": "Đào tạo",
        "isActive": true,
        "color": "#3498DB",
        "visibleRoles": ["lecture", "student"],
        "totalFiles": 10
      }
    ],
    "totalFiles": 25,
    "isActive": true,
    "isSystem": false,
    "createdAt": "2026-03-11T10:30:00.000000",
    "updatedAt": "2026-03-11T10:30:00.000000"
  }

RESPONSE 409 Conflict:
  { "detail": "Metadata type 'department' already exists" }
 
LUỒNG XỬ LÝ (WORKFLOW):
  1. Guard: require_auth.
  2. Kiểm tra trùng lặp key trong MongoDB.
  3. Lưu định nghĩa metadata mới (kèm allowedValues nếu có) vào DB.

────────────────────────────────────────────────────────────────────────────────
4.2. GET /api/metadata - Liệt kê Metadata Types
────────────────────────────────────────────────────────────────────────────────

MÔ TẢ:
  Liệt kê tất cả metadata type definitions.
  Kết quả được sắp xếp ưu tiên Metadata Hệ thống (`isSystem=true`) lên đầu, sau đó theo thời gian tạo.
  Authentication: Required (any valid user)

QUERY PARAMETERS:
  - isActive: bool (optional) - Lọc theo trạng thái isActive của metadata type.
  - keywords: string (optional) - Tìm kiếm theo displayName của cả key và value.

RESPONSE 200 OK:
  {
    "metadataTypes": [
      {
        "metadataId": "67a5f2c8abc123def456790",
        "key": "department",
        "displayName": "Phòng ban",
        ...
      }
    ],
    "total": 5
  }
 
LUỒNG XỬ LÝ (WORKFLOW):
  1. Guard: require_auth (bất kỳ user nào đã login).
  2. Nếu user không phải Admin: ép buộc isActive=true. Chỉ lấy Metadata Key đang hoạt động.
  3. Lọc Allowed Values: Nếu user không phải Admin, chỉ lấy các giá trị có isActive=true.
  4. Nếu user là Admin: cho phép dùng query isActive (true/false/không truyền).
  5. Nếu có keywords, tìm kiếm case-insensitive trong displayName của MetadataType và AllowedValue.
  4. Lọc lấy toàn bộ metadata types từ MongoDB.

4.8. DELETE /api/metadata/{key} - Xóa Metadata Type (Hard Delete)
────────────────────────────────────────────────────────────────────────────────

MÔ TẢ:

  CHỈ cho phép xóa nếu số lượng file đang dùng tag này (totalFiles) bằng 0.
  Không thể xóa nếu isSystem=true.

RESPONSE 204 No Content:
  (Thành công, không có body)
RESPONSE 409 Conflict:
  { "detail": "Cannot delete metadata type 'department' because it is in use by 5 files." }

RESPONSE 403 Forbidden:
    { "detail": "Cannot delete system metadata type: 'accessScope'. 'accessScope' is protected." }

RESPONSE 404 Not Found:
  { "detail": "Metadata type 'department' not found" }
 
LUỒNG XỬ LÝ (WORKFLOW):
  1. Guard: require_auth.
  2. Kiểm tra isSystem: Chặn xóa metadata hệ thống (403).
  3. Kiểm tra totalFiles: Chặn xóa nếu có file đang sử dụng metadata này (409).
  4. Hard delete bản ghi khỏi MongoDB.

  - `academicYear` / `cohort`: Cấm xóa giá trị `all`.

RESPONSE 200 OK: (MetadataTypeResponse đã cập nhật)

RESPONSE 409 Conflict:

RESPONSE 404 Not Found:
  { "detail": "Value 'xxx' not found in metadata type 'yyy'" }
 
LUỒNG XỬ LÝ (WORKFLOW):
  1. Guard: require_auth.
  2. Kiểm tra total_files của giá trị metadata cụ thể.
  3. Nếu total_files > 0: Từ chối xóa (409).
  4. Thực hiện xóa giá trị đó khỏi mảng allowedValues trong DB.

5.2. GET /api/files - Liệt kê Files
────────────────────────────────────────────────────────────────────────────────

QUERY PARAMETERS:
  - page           : (optional, int, default: 1)
  - limit          : (optional, int, default: 50)
  - fileStatus     : (optional, str) Filter by file status (active, processing, failed)
  - keywords       : (optional, str) Search by display_name (partial match)
  - metadataFilter : (optional, str/JSON) JSON filter string. 
    + **Multiple Values (OR - BẮT BUỘC)**: `{"cohort": ["K65", "K66"]}`, `{"academicYear": ["2024-2025"]}`
    + **Lưu ý**: Endpoint filter BẮT BUỘC phải truyền mảng (list) cho các value, nếu truyền Single Value (ví dụ: `{"cohort": "K65"}`) sẽ báo lỗi HTTP 400.
    + **Multiple Keys (AND)**: `{"cohort": ["K65"], "academicYear": ["2024-2025"]}`
    + **'all' value logic**: Tự động hỗ trợ giá trị "all" nếu được định nghĩa trong Metadata Type.

REQUEST EXAMPLE (Auth Required):
  GET /api/files?page=1&limit=20&metadataFilter={"cohort":["K65","K66"]}

  2. Lấy storeId mặc định qua resolve_store(None).

  4. Lọc MongoDB: Hỗ trợ query theo mảng hoặc giá trị đơn qua $in.

  1. Resolve default store (nếu bỏ trống storeId → gọi default store → error)
  2. Bắt buộc require_auth() → Lấy role từ JWT Header
  3. Enforce accessScope theo Auth Role:
     - student: Only "student" scope
     - lecture: Only "lecture" scope
     - admin: All scopes
  4. Gọi Gemini generate_content với File Search tool
  5. Extract sources từ grounding metadata
  6. Lấy presigned URLs cho mỗi source từ R2
  7. Return answer + sources + token_usage

Các endpoint FILES, CHAT, EMAIL DRAFT đều hỗ trợ auto store resolution:

Email draft KHÔNG phải là API endpoint.

                                                          ↓ yes
                                               InquiryService.process()

                      9b. gRPC CLIENTS (PYTHON-RAG → NEST-API)
════════════════════════════════════════════════════════════════════════════════

python-rag gọi nest-api qua gRPC theo hai luồng:

1. XÁC THỰC JWT (AuthService.VerifyToken)
   Proto:   app/proto/auth.proto    (package app_server)
   Stub:    app/proto/auth_pb2_grpc.py (AuthServiceStub — client only)
   Client:  app/services/integrations/grpc_client.py (GrpcClient)
   Dùng khi: Mọi request tới endpoint cần auth (require_auth / require_admin)
   Env:     GRPC_URL (default: localhost:5000)

   Quy trinh:
     Authorization: Bearer <token>
       → _extract_token() tách token
       → GrpcNestAuthClient.verify_token(token)
       → gRPC VerifyTokenRequest(token=token)
       → VerifyTokenResponse.payload (map<string,string>)
       → {sub, email, role, ...}
     require_admin: kiểm thêm role == "admin" → 403 nếu không phải

2. TẠO GMAIL DRAFT (InquiryService.Create)
   Proto:   app/proto/inquiry.proto   (package app_server)
   Stub:    app/proto/inquiry_pb2_grpc.py (InquiryServiceStub — client only)
   Client:  app/services/integrations/grpc_client.py (GrpcClient)
   Dùng khi: Sau khi Gemini soạn xong draft cho email inquiry
   Env:     GRPC_URL (dùng chung với auth)

   Request:  CreateEmailDraftRequest { messageId, draftSubject, draftBody }
   Response: CreateEmailDraftResponse { success }
   Graceful: Nếu nest-api trả UNIMPLEMENTED → log warning, trả False (không crash)

nest-api yêu cầu:
  - Chạy với START_GRPC=true
  - Port gRPC: 5000 (mặc định)


════════════════════════════════════════════════════════════════════════════════
                      10. RABBITMQ MESSAGE FORMAT
════════════════════════════════════════════════════════════════════════════════

NestJS (nest-api) publish message lên queue với pattern "ingested":

QUEUE NAME: Cấu hình qua RABBITMQ_INGEST_QUEUE (mặc định: "email_ingest_queue")
            nest-api dùng env RABBITMQ_QUEUE (mặc định: "email_ingest_queue")

MESSAGE FORMAT (NestJS RMQ transport):
  {
    "pattern": "ingested",
    "data": {
      "messageId": 123,           // int - ID trong DB của nest-api
      "subject": "Tiêu đề email", // string | null
      "senderEmail": "a@b.com",   // string | null (nullable trong Message entity)
      "senderName": "Nguyễn A",   // string | null (nullable trong Message entity)
      "content": "Nội dung..."    // string
    }
  }

python-rag consumer (RabbitMQ queue):
  - Nhận message → validate với IngestMessage schema
  - NẾU lỗi schema: basic_ack (bỏ qua, tránh poison loop)
  - NẾU xử lý thành công: basic_ack
  - NẾU xử lý thất bại: basic_nack + requeue=True (retry)
  - Timeout xử lý: 120 giây
  - Auto reconnect với exponential backoff (tối đa 10 lần, delay tối đa 60s)

METADATA FILTER FORMAT (dùng trong Chat và File Search):

  JSON format (request bắt buộc sử dụng mảng cho value):
    {
      "department": ["dao_tao"],
      "access_scope": ["student"]
    }

  Gemini filter string (sau khi convert):
    'department="dao_tao" AND accessScope="student"'

  List values (OR điều kiện - cùng key):
    Request: { "cohort": ["K20", "K21"] }
    Gemini/MongoDB:  '(cohort="K20" OR cohort="K21")'
    
  Multiple keys (AND điều kiện - khác key):
    Request: { "academicYear": ["2024"], "department": ["dao_tao"] }
    Gemini/MongoDB: 'academicYear="2024" AND department="dao_tao"'

  hasAllValue=true (metadata type có AllowedValue với value="all"):
    Request: { "department": ["dao_tao"] }
    Gemini:  '(department="dao_tao" OR department="all")'

Role-based accessScope (trong /api/chat/query và /api/chat/stream):
  - Hệ thống lấy role của người dùng qua JWT Auth Header (Authorization: Bearer <token>)
  - role="student": accessScope chỉ truy xuất được ["student"]
  - role="lecture": accessScope chỉ truy xuất được ["lecture"]
  - role="admin": thấy tất cả (bao gồm ["private"])
